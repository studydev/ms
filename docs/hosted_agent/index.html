1. 왜 Microsoft Foundry와 Hosted Agent가 등장했는가?

AI Agent의 부상과 현실적 한계

2024~2025년 사이 AI Agent는 단순 챗봇을 넘어 도구를 호출하고, 외부 데이터에 접근하며, 멀티-스텝 추론을 통해 자율적으로 작업을 완료하는 애플리케이션으로 진화했습니다. 그러나 대부분의 조직이 Agent를 직접 구축할 때 다음과 같은 공통된 문제에 부딪혔습니다.

기존 Agent 구현의 핵심 과제
  • 인프라 복잡성 — 컨테이너화, 웹 서버 설정, 오토스케일링, 롤백 파이프라인을 직접 설계·운영해야 합니다.
  • 상태 관리 — 대화 이력(Thread), 컨텍스트 윈도우, 메모리 영속성을 개발자가 직접 구현해야 하며, Redis·CosmosDB 등 별도 인프라가 필요합니다.
  • 보안 통합 — API 키 관리, 인증·인가(Entra ID/RBAC), 네트워크 격리, 콘텐츠 필터링을 각각 설정해야 합니다.
  • 도구(Tool) 오케스트레이션 — 코드 인터프리터, 파일 검색(RAG), Function Calling 등을 별도 서비스로 구축하고 연동해야 합니다.
  • 관찰성(Observability) 부재 — Agent의 의사결정 과정을 추적·디버깅할 End-to-End 트레이싱 체계가 없어 "블랙박스" 문제가 발생합니다.
  • 멀티 에이전트 조율 — 여러 Agent 간 협업이 필요할 때, 라우팅·핸드오프·공유 컨텍스트를 직접 구현해야 합니다.

Microsoft Foundry Agent Service — 플랫폼 수준의 해결

Microsoft Foundry Agent Service(이전 Azure AI Foundry Agent Service)는 이러한 과제를 플랫폼 수준에서 해결하기 위해 등장했습니다. 핵심 아이디어는 다음과 같습니다.

  • Fully Managed Runtime — 호스팅, 스케일링, 대화 오케스트레이션, 라이프사이클 관리를 플랫폼이 담당합니다.
  • Built-in Enterprise Security — Microsoft Entra ID, Azure RBAC, 콘텐츠 필터, VNet 격리가 기본 내장되어 있습니다.
  • Built-in Tools — Web Search, File Search, Code Interpreter, Memory, MCP 서버, Custom Functions이 관리형으로 제공됩니다.
  • Multi-model Support — GPT-4o, GPT-4.1, Llama, DeepSeek 등 Foundry 모델 카탈로그의 다양한 모델을 코드 변경 없이 교체할 수 있습니다.
  • End-to-End Observability — 모든 모델 호출, 도구 실행, 의사결정을 추적하는 에이전트 트레이싱과 Application Insights 연동이 기본 제공됩니다.

Hosted Agent — "Bring Your Own Code"의 완성

Foundry Agent Service는 세 가지 에이전트 유형을 지원합니다: Prompt Agent(노코드), Workflow Agent(선언적 멀티-에이전트 오케스트레이션), 그리고 Hosted Agent.

Hosted Agent는 개발자가 원하는 프레임워크(Microsoft Agent Framework, LangGraph, 또는 커스텀 코드)로 작성한 에이전트를 컨테이너로 패키징하여 Agent Service에 배포하는 방식입니다. 개발자는 오케스트레이션 로직에 집중하고, 인프라·보안·스케일링은 Foundry가 관리합니다.

Hosted Agent가 해결하는 것

오픈소스 프레임워크로 에이전트를 구축할 때 일반적으로 관리해야 하는 — 컨테이너화, 웹 서버 설정, 보안 통합, 메모리 영속성, 인프라 스케일링, 데이터 전송, 계측, 버전 롤백 — 이 모든 것을 Hosted Agent 플랫폼이 대신합니다.

핵심 차별점

Hosted Agent의 본질적 가치는 "개발자는 에이전트 로직(What)에만 집중하고, 플랫폼이 운영(How)을 책임진다"는 것입니다. 기존 방식에서 전체 노력의 60~70%를 차지하던 인프라·보안·관찰성 업무가 플랫폼으로 이전됩니다.

3. 아키텍처 개요

Hosted Agent의 구조적 동작 원리

Hosted Agent는 개발자 코드를 컨테이너 이미지로 패키징하여 Foundry Agent Service의 관리형 런타임 위에서 실행하는 구조입니다. 아래의 네 가지 핵심 계층이 유기적으로 협력합니다.

🛠
[ 아키텍처 다이어그램 삽입 위치 ]
아래 설명된 4개 계층(Model → Tools → Orchestration → Runtime)의 관계를 시각화하는 다이어그램을 삽입하세요.
권장 형식: Foundry Portal 스크린샷 또는 Visio/Mermaid 기반 커스텀 다이어그램

계층 1 — Models (추론 엔진)

Foundry 모델 카탈로그에서 GPT-4o, GPT-4.1, Llama, DeepSeek 등 다양한 LLM을 선택할 수 있습니다. 모델은 Agent의 추론(reasoning)과 언어 생성을 담당하며, 코드 변경 없이 모델을 교체할 수 있습니다. Model Router(Preview)를 사용하면 요청별로 비용/성능 최적 모델을 자동 선택하는 것도 가능합니다.

계층 2 — Tools (실행 능력)

Agent가 외부 세계와 상호작용하는 수단입니다. Foundry Agent Service는 다음과 같은 Built-in 도구를 제공합니다.

도구 역할
Web Search실시간 인터넷 검색을 통한 최신 정보 접근
File Search벡터 스토어 기반 관리형 RAG — 문서 업로드 후 자동 청킹·검색·인용
Code Interpreter샌드박스 환경에서 Python 코드 실행 (데이터 분석, 차트 생성)
Memory대화 간 지속적 컨텍스트 유지 (Preview)
MCP ServersAzure DevOps 등 외부 서비스와의 표준화된 연결
Custom Functions개발자 정의 비즈니스 로직 호출 (Function Calling)

모든 도구는 관리형 인증(Service Managed Credentials, On-Behalf-Of/OBO)을 지원하여 별도 자격 증명 관리가 불필요합니다.

계층 3 — Orchestration (오케스트레이션)

Hosted Agent에서 오케스트레이션 로직은 개발자가 코드로 작성합니다. Agent Framework, LangGraph, 또는 커스텀 코드를 사용하여 도구 호출 순서, 멀티-스텝 추론, 에이전트 간 협업 로직을 정의합니다.

Hosting Adapter는 이 개발자 코드와 Foundry 런타임 사이의 추상화 계층입니다. Foundry 요청/응답 포맷 ↔ 프레임워크 네이티브 데이터 구조 간 자동 프로토콜 변환, 대화 관리, 스트리밍 이벤트 생성, OpenTelemetry 기반 관찰성 연동을 처리합니다.

계층 4 — Agent Runtime (런타임)

Foundry의 관리형 런타임이 모든 운영 측면을 담당합니다.

  • 프로비저닝 & 오토스케일링 — CPU/Memory 레플리카 크기 설정 후 자동 스케일링
  • 대화 오케스트레이션 & 상태 관리 — Thread, Run, 메시지를 플랫폼이 관리
  • ID 관리 — 미배포 Agent는 프로젝트 Managed Identity, 배포 후에는 전용 Agent Identity 할당
  • 관찰성 — Application Insights 통합, 에이전트 트레이싱, 메트릭 대시보드
  • 버전 관리 & 배포 — 자동 스냅샷, 롤백, 안정적 엔드포인트 제공
🔄
[ 요청 처리 시퀀스 다이어그램 삽입 위치 ]
사용자 → Agent Runtime → Model → Tool 호출 → 응답 반환의 흐름을 시퀀스 다이어그램으로 삽입하세요.
특히 Hosting Adapter가 중간에서 프로토콜 변환을 수행하는 과정을 표현하면 이해에 도움됩니다.
Agent 유형별 오케스트레이션 차이
  • Prompt Agent — 오케스트레이션을 플랫폼이 자동 처리 (노코드)
  • Workflow Agent — YAML 선언형 정의로 멀티-에이전트 분기/합류 오케스트레이션
  • Hosted Agent — 개발자가 코드로 직접 오케스트레이션 (최대 유연성)

4. Hands-on Lab: Deploy Your First Hosted Agent

이 실습에서는 Azure Developer CLI(azd)를 사용하여 Web Search 도구를 갖춘 Hosted Agent를 Foundry Agent Service에 배포합니다. 실습 완료 후, Foundry Playground에서 Agent와 대화할 수 있습니다.

Hosted Agent는 현재 Public Preview 상태입니다.

배포된 Agent는 과금이 발생합니다. 실습이 끝나면 반드시 리소스를 정리하세요.

새 프로젝트를 초기화하면 대화형 설정이 시작됩니다.

azd ai agent init

프롬프트에서 "Start new from a template"을 선택한 후, 아래 항목을 순서대로 설정합니다.

설정 항목설명예시 값

MCP 서버를 사용하지 않는 경우

  value: <CONNECTION_ID_PLACEHOLDER>

2 로컬에서 Agent 테스트 (Test the agent locally)

2-1. Agent 로컬 실행

이 명령은 자동으로 환경 설정, 종속성 설치, Agent 시작을 처리합니다.

azd ai agent run

azure.yaml에 정의된 startupCommand를 사용하여 Agent를 로컬(localhost:8088)에서 시작합니다.

📷 [ 스크린샷 삽입 위치 ]
azd ai agent run 실행 후 터미널에 Agent가 시작된 모습 캡처

2-2. 테스트 메시지 전송

별도의 터미널을 열어 아래 명령으로 Agent에 테스트 메시지를 보냅니다.

# 별도 터미널에서 실행
azd ai agent invoke --local "What is Microsoft Foundry?"
로컬 테스트의 장점
  • 컨테이너화 전에 Agent 동작을 빠르게 검증할 수 있습니다.
  • 개발 환경에서 디버깅이 용이합니다.
  • 다양한 입력 시나리오를 빠르게 테스트할 수 있습니다.
  • Foundry Responses API와의 호환성을 사전 검증할 수 있습니다.

일반적인 로컬 실행 오류 해결

오류해결 방법
AuthenticationError / DefaultAzureCredential failure azd auth login으로 세션 갱신
ResourceNotFound Foundry Portal에서 엔드포인트 URL이 맞는지 확인
DeploymentNotFound Build > Deployments에서 배포 이름 확인
Connection refused 포트 8088을 사용 중인 다른 프로세스가 없는지 확인
Web Search 결과를 기반으로 Microsoft Foundry에 대한 정보를 포함한 응답이 표시됩니다.
3 Foundry Agent Service에 배포 (Deploy to Foundry)

로컬 테스트가 성공적으로 완료되면, Agent를 Foundry Agent Service에 배포합니다.

azd deploy

배포 완료 후 다음과 같은 출력이 표시됩니다.

Deploying services (azd deploy)

(✓) Done: Deploying service af-agent-with-foundry-tools

  - Agent playground (portal):
    https://ai.azure.com/nextgen/.../build/agents/af-agent-with-foundry-tools/build?version=1

  - Agent endpoint:
    https://ai-account-<name>.services.ai.azure.com/api/projects/<project>/agents/af-agent-with-foundry-tools/versions/1
📷 [ 스크린샷 삽입 위치 ]
azd deploy 완료 후 터미널에 표시되는 Agent Playground URL과 Endpoint URL 캡처

배포 후 상태 확인

# Agent 상태 확인
azd ai agent show

# Agent 로그 확인
azd ai agent monitor
azd ai agent show 실행 시 Agent 상태가 "Started"로 표시됩니다. Foundry Portal과 REST API 엔드포인트 URL이 제공됩니다.
4 Foundry Playground에서 검증

방법 A: CLI로 테스트

# 배포된 Agent에 메시지 전송
azd ai agent invoke "What is Microsoft Foundry?"

방법 B: Foundry Portal Playground에서 테스트

  1. Foundry Portal에 Azure 계정으로 로그인합니다.
  2. Recent projects에서 프로젝트를 선택하거나, All projects에서 찾습니다.
  3. 좌측 내비게이션에서 Build → Agents를 선택합니다.
  4. Agent 목록에서 배포한 Agent 이름을 클릭합니다.
  5. 상단 툴바의 "Open in playground"를 클릭합니다.
  6. 채팅 인터페이스에서 "What is Microsoft Foundry?"를 입력하고 Enter를 누릅니다.
  7. Web Search 결과를 기반으로 Agent가 응답하는지 확인합니다.
📷 [ 스크린샷 삽입 위치 ]
Foundry Portal Playground에서 Agent와 대화하는 화면 캡처.
Agent가 Web Search 결과를 포함하여 응답하는 모습이 보여야 합니다.
Playground에서 Agent가 Web Search 도구를 사용하여 Microsoft Foundry에 대한 최신 정보를 포함한 응답을 생성합니다. 응답에 소스 정보가 포함될 수 있습니다.
5 리소스 정리 (Clean up)
과금 방지를 위해 반드시 리소스를 정리하세요.

이 명령은 리소스 그룹 내 모든 Azure 리소스(Foundry Project, 모델 배포, Container Registry, Application Insights, Hosted Agent)를 영구 삭제합니다. 이 작업은 되돌릴 수 없습니다.

5-1. 삭제 대상 미리보기

# 삭제될 리소스를 미리 확인
azd down --preview

5-2. 리소스 삭제 실행

azd down

5-3. 삭제 확인

Azure Portal에서 리소스 그룹으로 이동하여 리소스가 더 이상 표시되지 않는지 확인합니다. 리소스 그룹이 비어 있다면 그룹 자체도 삭제할 수 있습니다.

모든 Azure 리소스가 성공적으로 삭제되고, Azure Portal에서 해당 리소스 그룹이 비어 있거나 삭제된 상태로 확인됩니다.

5. 주요 설정 및 개념 정리

핵심 파일 구조

파일역할
agent.yaml Agent 매니페스트 — 모델 배포, 도구 연결, 환경 변수 등 Agent의 전체 구성을 선언적으로 정의합니다.
azure.yaml Azure Developer CLI 서비스 정의 — Agent를 azd 서비스로 등록하고, startupCommand, 배포 설정을 관리합니다.
main.py Agent 애플리케이션 진입점 — Agent Framework/LangGraph로 작성된 에이전트 로직과 Hosting Adapter 초기화 코드가 포함됩니다.
Dockerfile 컨테이너 이미지 빌드 정의 — --platform linux/amd64 지정이 필수입니다 (Hosted Agent는 Linux AMD64에서 실행).
requirements.txt Python 종속성 — azure-ai-agentserver-agentframework 등 Foundry SDK 패키지가 포함됩니다.

주요 환경 변수

변수명설명
PROJECT_ENDPOINT Foundry Project의 API 엔드포인트 URL (예: https://<resource>.services.ai.azure.com/api/projects/<project>)
MODEL_DEPLOYMENT_NAME 사용할 모델 배포 이름 (예: gpt-4.1)
